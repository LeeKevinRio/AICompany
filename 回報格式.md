# 回報格式（每一次回應都必須遵守）

> 這份檔案定義「每一次回應」對 CEO 的標準回報格式。
> CLAUDE.md 第 0 節已把本檔列為**最高原則之一**：每一次執行、每一個 session 問題的回應，都必須走過這個格式。
> 目的：CEO 每次都能一眼看到——**明確結論**、**每位主管的回報（即使沒做事也要回報）**、**要怎麼驗證**、**下一步做什麼**。

---

## 每一次回應必含的四段

### 1. 本次結論
- 用一兩句把「這次執行 / CEO 問的問題」講出結論：**做完了 / 卡住了 / 需要 CEO 決策**，以及最關鍵的一句話。
- 每一次都要有結論，沒有「做到一半沒交代」這種事。

### 2. 各部門回報（有參與的每位一行；其餘一行彙總）
部門清單以 [`docs/org-chart.md`](docs/org-chart.md) 為準。回報方式：

- **有參與的部門**：每位一行，寫「做了什麼」（qa-reviewer 要含 Codex 第二意見結果）。
- **未參與的部門**：最後用一行彙總，例如「其他部門（product-manager、art-lead、…）：本次未參與（純文件任務）」。
- 若本次任務**應該**有某部門把關卻沒有（例如建議類文案沒過 risk-compliance-officer），必須明寫並說明原因。

> 用意：讓 CEO 一眼看出哪些角色被動到、哪些沒被動到，避免有人默默被略過，也避免該把關的關卡被跳過。

### 3. 驗證
- 這次的成果**要怎麼確認是對的**？
- 需要 **CEO 親自驗證**什麼？（例如：把 app 跑起來看畫面、確認某個決策方向、檢查某份輸出）
- 如果有自動化驗證（測試、build、Codex review），寫出實際結果。

### 4. 下一步
- 接下來**具體要做什麼**（一到三點），或目前**在等 CEO 的哪個決定**。

---

## 規則備註

- **隨規模縮放**：小回應每段一兩行即可；沒參與的主管一行帶過，不要灌水。重點是四段都在、結論清楚。
- **誠實回報**：失敗就說失敗、跳過就說跳過、誤判就更正，不得粉飾。
- **語言**：繁體中文（台灣用語），技術名詞與 code 保留英文（同 CLAUDE.md 第 0 節）。
